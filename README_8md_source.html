1 # The IDA-LET-Machine (ILM)
2 The ILM is a simple implementation of the logical execution time (LET) paradigm for the automotive domain.
3 In general it is based on a double buffering, to implement a zero-time communication between different cores.
4 You can find the doxygen generated documentation [here](https://matthiasb85.github.io/IDA-LET-Machine/index.html "ILM documentation")
5 
6 The ILM was developed as an operating system (OS) and hardware independent module.
7 At the moment, we support the standard OSEK/AUTOSAR OS API, as the first OS port was based on ERIKA OS.
8 The current hardware port is based on the CCU6 module of the AURIX TC2xx which is usually used for PWM generation.
9

11 Network Engineering, abbreviation **IDA**) of the TU Braunschweig.
12 During 2016 and 2017 the development was mainly influenced by a collabortaion between IDA and Daimler RD/EIS.

14 ## Where is the ILM intended for?
15 Our mission is, to provide a simple and efficient test framework for the LET paradigm.
16 This framework can be used in academia (e.g. for evaluation) or also for prototyping.
17 In general the ILM can be used everywhere, where you would like to test something new, but don't want to have a huge ammount of 3rd party software in the background.
18 
19 ## Where is the ILM not intended for?
20 Although the license model does not exclude it explicitly, we would recommend not to use the ILM in any kind of series implementation.
21 We have written the software to the best of our knowledge and belief, nevertheless it may still contain some bugs.
22 Therefore we can not provide any kind of guarantee for functional correctness.
23 For this kind of integegration there are several other commercial partners who provide their own implementation.
24 As an example, Bosch and Continental do both provide an implementation of their interpretation of the LET paradigm.

31 ## Where to start?
32 In order to test the example, you need a build environment for the AURIX microcontroller.
33 We use a Linux in combination with the HighTech tricore-gcc, but a cygwin installation (as used for ERIKA OS on a Windows) should also do the work.
34 
35 ```
36 git clone https://github.com/matthiasb85/IDA-LET-Machine.git
37 cd ./IDA-LET-Machine
38 cd Example
39 make
40 ```
41 The Makefile checks if the 3rdParty software is available, if this is not the case the ./3rdParty/Setup3rdParty.sh script is executed.
42 The script downloads the corresponding standalone version of ERIKA OS and RT-Druid, and applies some patches on it.
43 The hardware port for the AURIX microcontroller uses Infineons iLLD library, which unfortunately is only available after registration.
44 If the library is not available in ./3rdParty/Download when ./3rdParty/Setup3rdParty.sh is called, the script reports:
45 
46 ```
47 Can't find Infineon iLLD!
48  1.) Please get in contact with Infineon
49  2.) Download the AURIX_iLLD_Package_TC2xx_v1.0.1.3.0 driver package
50  3.) Copy AURIX_iLLD_Package_TC2xx_v1.0.1.3.0.zip to ./3rdParty/Download
51  4.) Run 'make'/'make prepare' in ./Example or Setup3rdPartySoftware.sh in ./3rdParty
52 ```
53 An important constraint is, that the entire library is available as source code.
54 The reason for this is a bug inside the official CCU6 driver of the iLLD (which is patched during ./3rdParty/Setup3rdParty.sh).

66 ## Application example
67 The application consist of 18 LET tasks.
68 The layout of the example application can be found in ./Example/ilm_example.ods.
69 The example uses a setup with three application cores.
70 As the used AURIX microcontroller provides three cores, we use core C0 to generate the needed IRQ pattern.
71 In addition, the application tasks are executed on all three cores.
72 The worst-case execution time (WCET) of the corresponding block is assigned to an LET task on both application cores.
73 If no function block is assigned to an LET task on C0, C1 or C2, the corresponding WCET is zero.
74 In order to simulate an accurate timing behavior of the function blocks, we used an optimized **nop** loop consuming CPU time.
75 Due to the different periods of the system, the hyperperiod is 100ms.
76 
77 During execution of the simulated function blocks, we use pin toggles to indentify the current states.
78 Those pin toggles are issued through the ILM hooks and at execution begin/end of a function block.
79 The physical pin magging is located in ./Example/IO/gpio.c and the hook functions in ./Example/ILM/ilm_hooks.c
80 
81 If you use a Lauterbach Trace32 based setup, you can use the ''make debug'' command inside ./Example to start a debug session.
82 With ''t32_folder'', ''t32_arch'' and ''addr'' you can pass Trace32 base folder, host CPU architecure and IP address (DNS name) of your trace hardware to the system.
83 If you omit the ''addr'' parameter, Trace32 uses the USB interface and searches for a locally connected device.
84 As an example, the ''make debug'' call on my PC looks like this:
85 ```
86 make debug t32_folder=/home/matthiasb/opt/t32/files t32_arch=pc_linux addr=lauterbach1
87 ```

115  * **LET IRQ/Event:** Set to 1 when processing an LET IRQ/Event. The AURIX HW port generates an CCU6 IRQ on C0 and redirectes the event to C1/2 via a general purpose service request (GPSRN).
116  * **Start LET task (y):** Set to 1 if an LET task with period *y* (100/20/10) is activated. Set to 0 at the end of an LET task. As the LET tasks are sequenced back-to-back, only a small peak is visible between LET task with the same period *y*.
117  * **WCET function block (y):** Set to 1 when the execution of the corresponding function block inside the corresponding OS task starts. Set to 0 when the block function finishes its execution. Again *y* denotes the corresponding period. Measuring the resulting pulse should lead to a value close to the defined worst-case execution time (WCET) defined in app_core_[0,1,2].c
